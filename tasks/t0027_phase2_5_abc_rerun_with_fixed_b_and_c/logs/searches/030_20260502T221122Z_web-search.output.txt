    "Based on the search results, I found relevant information about this Django ASGI middleware issue:\n\n## Django ASGI Middleware Coroutine Issue\n\nWhen using ASGI (such as with uvicorn), the first middleware according to the list in settings.py receives a coroutine as its response parameter, while all other middlewares down the line receive a django.http.response.HttpResponse object.\n\nThis was identified as Django issue #31928. The issue can be reproduced by creating a dummy class-based middleware that prints the types of arguments it receives in its process_response method, which shows output like: `<class 'django.core.handlers.asgi.ASGIRequest'> <class 'coroutine'>`\n\n### Root Cause\n\nSecurityMiddleware and the three cache middlewares were not calling super().init() during their initialization or calling the required MiddlewareMixin._async_check() method. This made the middlewares not properly present as coroutine and confused the middleware chain when used in a fully async context.\n\n### Solution\n\nThe issue was fixed by ensuring proper detection of an async get_response in various middlewares. To avoid this problem, ensure that custom middlewares:\n\n1. Call `super().__init__()` in their `__init__` method\n2. Call the required `MiddlewareMixin._async_check()` method \n3. Properly mark themselves as coroutine functions when needed using `asgiref.sync.markcoroutinefunction`\n\nThis ensures the middleware chain properly handles async contexts in ASGI deployments."
